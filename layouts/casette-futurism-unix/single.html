Designing for Failure

2024-12-01 15K -rw-r--r--

Three years of running distributed systems at scale taught me one uncomfortable truth: your system will fail. The question isn't if, but how gracefully. Every architecture diagram I drew in year one assumed happy paths. Every post-mortem from year two proved those assumptions wrong.

This post isn't theoretical. These are patterns extracted from real incidents — the 3 AM pages, the cascading failures, the moments when "five nines" felt like a marketing fiction rather than an engineering target.

The Uptime Illusion

We spent our first year chasing 99.999% availability. It sounds impressive on a slide deck. In practice it means less than 5 minutes of downtime per year. We hit that target exactly once — the quarter we got lucky.

The problem wasn't our code. It was our mental model. We treated failures as exceptions to be eliminated rather than constants to be managed. Every dependency was a single point of failure waiting to activate.

The most reliable distributed system isn't the one that never fails. It's the one where failures are boring, expected, and handled automatically.

Pattern: Circuit Breakers

Borrowed from electrical engineering. When a downstream service starts failing, stop calling it. Give it time to recover. Serve degraded responses instead of cascading the failure upstream.

class CircuitBreaker:
    def __init__(self, threshold=5, timeout=60):
        self.failures = 0
        self.threshold = threshold
        self.timeout = timeout
        self.state = "closed"
        self.last_failure = None

    def call(self, fn, *args, **kwargs):
        if self.state == "open":
            if time.time() - self.last_failure > self.timeout:
                self.state = "half-open"
            else:
                return self._fallback()

        try:
            result = fn(*args, **kwargs)
            self._on_success()
            return result
        except Exception:
            self._on_failure()
            return self._fallback()

    def _on_failure(self):
        self.failures += 1
        self.last_failure = time.time()
        if self.failures >= self.threshold:
            self.state = "open"

    def _on_success(self):
        self.failures = 0
        self.state = "closed"

The key insight: _fallback() doesn't mean "show an error." It means "return the best answer you can without this dependency." Cached data. A default. A simpler computation. Anything that keeps the user's request moving.

Pattern: Bulkheads

Borrowed from ship design. A hull breach in one compartment doesn't sink the entire vessel because bulkheads contain the flooding. Apply the same principle to your services:

  • Separate thread pools per downstream dependency — one slow service can't exhaust all your connections
  • Separate connection pools for reads vs. writes — a write storm doesn't block read traffic
  • Separate deployment units for critical vs. non-critical paths — your checkout flow shouldn't share fate with your recommendation engine

Pattern: Timeout Budgets

Every request gets a total time budget. As it passes through each service in the chain, the remaining budget shrinks. When the budget runs out, you stop making downstream calls and return what you have.

class TimeoutBudget:
    def __init__(self, total_ms):
        self.deadline = time.monotonic() + (total_ms / 1000)

    @property
    def remaining_ms(self):
        left = self.deadline - time.monotonic()
        return max(0, int(left * 1000))

    @property
    def exhausted(self):
        return self.remaining_ms == 0

Without timeout budgets, a single slow service at the bottom of your call graph can cause every service above it to hold connections open, consuming resources all the way up. With budgets, the damage is bounded.

What Actually Matters

After three years, I've stopped optimizing for Mean Time Between Failures. I optimize for Mean Time To Recovery. The difference is philosophical:

  • MTBF thinking: "How do we prevent this from ever happening?"
  • MTTR thinking: "When this happens, how fast do we recover?"

MTTR thinking leads to better investment. Instead of over-engineering prevention for scenarios you can't predict, you build robust detection, automated recovery, and graceful degradation. Your system breaks in small, contained ways instead of spectacular, correlated ones.

Build your circuit breakers. Install your bulkheads. Set your timeout budgets. Then practice failing — in staging, in production, on purpose. The team that's bored by outages is the team that recovers in minutes instead of hours.
