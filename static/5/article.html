Why I Still Write Raw SQL in 2026

Object-Relational Mappers (ORMs) like SQLAlchemy and Django ORM have revolutionized how we interact with databases in Python. They provide incredible productivity boosts for standard CRUD operations and help prevent basic SQL injection vulnerabilities.

But let's be honest: there's a point where the abstraction leaks. Hard.

The Breaking Point

When you need to perform complex analytical queries, hierarchical data aggregations, or leverage specific features of your database engine (like Postgres CTEs or Window Functions), wrestling with ORM syntax often takes longer than just writing the SQL.

Consider this nightmare ORM query for a rolling average:

# SQLAlchemy code that makes my eyes bleed
stmt = select(
    sales.c.date,
    func.avg(sales.c.amount).over(
        order_by=sales.c.date,
        rows=( -6, 0 )
    ).label('rolling_7_day_avg')
).group_by(sales.c.date)

It works, sure. But it requires mental gymnastics to translate from the relational algebra in your head to Python method chaining, and back to the SQL that actually gets executed.

Embracing the Raw String

Instead, I've found immense value in keeping my SQL skills sharp and utilizing raw SQL execution for complex reads. It's explicitly clear what is happening.

-- The same logic, but readable by any data engineer
SELECT 
    date,
    AVG(amount) OVER(
        ORDER BY date 
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ) as rolling_7_day_avg
FROM sales;

By keeping my simple operations in the ORM and pulling complex logic into raw SQL (executed securely with parameterized queries, naturally), I get the best of both worlds: cursor.execute(sql, params).

Don't be afraid to drop down a layer. Your database is a powerful engine—stop treating it like a dumb key-value store.
